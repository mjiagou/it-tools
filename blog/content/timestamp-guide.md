在计算机的世界里，时间不是“2026年1月11日”，而是一个不断增长的整数。这就是 **Unix 时间戳 (Unix Timestamp)**。

它定义为从 **1970年1月1日 00:00:00 UTC** 到现在所经过的秒数。尽管它看起来很简单，但在实际开发中，无数的 Bug 都源于对它的误解。

## 痛点一：秒 (s) 还是毫秒 (ms)？

这是新手最容易踩的坑。
* **PHP / Python / MySQL**：通常默认使用 **10位** 整数（精确到秒）。例如：`1768176000`
* **Java / JavaScript / Go**：通常使用 **13位** 整数（精确到毫秒）。例如：`1768176000000`

如果你在 JavaScript 里直接处理后端传来的 10 位时间戳，你会发现时间回到了 1970 年（因为数值太小了）。

## 痛点二：时区的魔术

记住：**Unix 时间戳本身是不带时区的**。它永远是相对于 UTC (格林威治标准时间) 的。

为什么你在北京看到的是 8 点，而在伦敦看到的是 0 点？那是“显示层”的事情。
* 数据库存储：建议永远存 UTC 时间戳。
* 前端展示：根据用户浏览器的本地时区进行格式化。

## 痛点三：2038 年问题

传统的 Unix 时间戳是使用 `int32`（32位有符号整数）存储的。它的最大值是 `2147483647`。
这个数字对应的时刻是 **2038年1月19日 03:14:07**。

过了这一秒，时间就会溢出，变成负数（1901年）。这就是著名的“2038 年问题”。现在的 64 位系统已经解决了这个问题，但维护老旧系统时仍需警惕。
